# Verification Before Completion

## 核心原则

**先验证，再声称完成。永远是证据在前，断言在后。**

## 铁律

```
没有验证过的代码，不能说"已修复"或"已完成"。
```

## 验证流程

```
在声称任何状态之前：

1. 确认：最终代码是什么样的？
2. 检查：所有修改点是否一致？
3. 审查：有没有遗漏的引用、遗留的旧变量？
4. 确认后才说"已完成"

跳过任何一步 = 没验证
```

## 常见问题

| 声称 | 需要确认 | 不算验证 |
|------|---------|---------|
| Bug 已修复 | 相关代码逻辑正确、无遗漏 | "应该没问题" |
| 代码改完了 | 所有相关文件都已修改 | 只改了一个文件 |
| 变量已删除 | 全局搜无残留引用 | "我记得删了" |
| 逻辑正确 | 逐行推演过数据流 | "大概是对的" |
| 需求满足 | 逐条对照用户要求 | "看起来差不多" |

## 嵌入式专项验证

修改完成后必须检查：
- 修改的变量/函数在其他文件是否有引用？
- 中断处理函数中是否引用了被删除/修改的变量？
- 状态机各状态转换是否仍然完整？
- 新增的 static 变量是否正确初始化？
- 条件判断的位运算是否与传感器映射一致？

## 红旗 — 立即停止

- 用"应该"、"可能"、"大概"
- 在确认前就表达满意（"好了！"、"搞定了！"）
- 想着"这次就算了"
- **任何暗示成功的措辞，但没有实际验证**
