# Systematic Debugging (嵌入式)

## 核心原则

**永远先找根因，再尝试修复。只修表象 = 失败。**

## 铁律

```
没完成 Phase 1（根因分析），不得提出修复方案。
```

## 何时使用

适用于任何技术问题：
- 小车行为异常（乱转、不停、跑偏）
- 传感器数值异常
- 状态机卡死或跳转错误
- 计时/计数错误
- 编译错误

**特别注意的触发场景：**
- "小改动应该就行的"的时候
- 已经试了好几次还没修好
- 上次的修复不管用
- 你不完全理解问题

## 四阶段流程

### Phase 1: 根因调查

**修复前必须先做：**

1. **仔细阅读错误现象**
   - 完全理解用户描述的异常行为
   - 不要跳过任何细节
   - 追问不明确的地方

2. **分析相关代码**
   - 读取所有相关文件和函数
   - 追踪变量从哪来、到哪去
   - 检查条件判断是否正确
   - 检查时序逻辑（中断、定时器、标志位）

3. **检查近期改动**
   - 哪些代码最近被修改了？
   - 改动之间有没有关联性？

4. **嵌入式特有的排查**
   - 传感器数据是否合法？
   - 中断是否被正确处理？
   - 变量是否在中断和主循环间正确共享（volatile）？
   - 计时器/计数器是否正确递增？
   - 状态机是否卡在某个状态？
   - 标志位是否被正确设置/清除？

### Phase 2: 模式分析

1. **找可对比的正常代码**
   - 有没有类似的逻辑是正常工作的？
   - 差异在哪里？

2. **理解整体逻辑**
   - 完整阅读相关状态机/数据流
   - 画出数据流向

3. **列出所有差异**
   - 哪怕看起来不重要也要列出来

### Phase 3: 假设和验证

1. **提出单一假设**
   - 明确："我认为 X 是根因，因为 Y"
   - 不要模糊

2. **最小改动验证**
   - 只改一处来验证假设
   - 不要一次改多个地方

3. **验证后再继续**
   - 改对了 → 进入 Phase 4
   - 没改对 → 提出新假设
   - 不要在上一个修复上叠加更多修复

4. **如果不知道**
   - 诚实说"我不理解 X"
   - 向用户询问更多信息

### Phase 4: 实施修复

1. **针对根因修复**
   - 只改真正导致问题的地方
   - 一次改一处
   - 不要顺便重构

2. **验证修复**
   - 向用户描述修复内容
   - 让用户在实际硬件上验证

3. **如果修复无效**
   - 停止！数一数已经试了几次
   - 少于 3 次：回到 Phase 1 重新分析
   - 达到 3 次：质疑架构设计，和用户讨论是否需要重新设计

## 红旗警示 — 立即停止

如果发现自己在想：
- "先随便改改试试"
- "应该是 X 的问题，直接改吧"
- "多个地方一起改，更快"
- "不太确定但应该没问题"
- "再试最后一次"（已经试了 2+ 次）

**全部意味着：停下来，回到 Phase 1。**
